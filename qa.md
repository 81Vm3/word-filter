 - 我们用元数据的方式表达某个词的严重程度，严重程度是一个整数，在一个与字典树并列的map里面存放 “metas map[int]Metadata”
 - 在本地存储时，用txt存储敏感词
line1 =ccount n；接下来 n 行为词；再接下来 n 行为对应严重系数。实现时按"逐行读取，先读 n，再读 n 个词，再读 n 个数字"，对 off-by-one 表述天然容忍。

2. 字典树节点怎么设计的
 - 终结节点存 wordID int + runeLen int；非终结 wordID = -1
 - 元数据 集中存在 Trie.metas map[int]Metadata（按 wordID 索引），便于 Update/Delete 不动 Trie 结构

6. 为什么去重表 words map[string]int（归一化后的词 -> wordID）要用 map，而不是直接在字典树上做去重？
 - 核心是职责分离：Trie 负责前缀结构和匹配；map 负责“完整词是否存在以及对应哪个 wordID”
 - 去重是完整 key 查询，map 均摊 O(1)，实现也更直接
 - CRUD（尤其 Update/Delete/Get）经常需要先按词精确拿到 wordID，map 可以一步定位
 - AC 匹配结果天然围绕 wordID，维护单独词 -> ID 索引能让词和匹配结果的关联更稳定
 - 不是 Trie 不能做去重，而是把去重索引独立成 map 在工程上更清晰、维护成本更低

7. 流式文本是怎么处理的？在 AC 自动机基础上改了什么？
 - 实际流程是“先建好自动机，再持续喂数据块”：会话启动时先确认自动机是最新的，然后把当前位置放在根节点，准备进入增量处理
 - 每次收到新 chunk，不是直接扫，而是先和上次残留的半包字节拼起来；这样可以正确处理 UTF-8 被截断在块边界的情况
 - 进入扫描后按字符推进状态机：每读到一个字符就做同样的归一化，再继续 AC 状态转移
 - 一旦命中，不只记“命中了什么词”，还同步记录全局位置（字符下标和字节下标）跨 chunk 的累计偏移，所以后续结果能直接映射回原始输入
 - 结果输出采用“累计视图”思路：随时可以拿当前总结果查看命中和替换文本，但会话本身不自动清空，后续追加会在已有状态上继续

8. 怎么把 AC 自动机从一次性函数变成可增量驱动的？（侧重状态维护）
 - 关键是把“一次性扫描时临时存在栈里的变量”变成“可长期保存的会话状态”，这样每次只处理新输入，不必从头跑
 - 自动机本体不变，变的是执行模型：规则仍然是同一套状态转移，但状态不在函数内部结束即销毁，而是持续保留到下一次输入
 - 会话至少维护四类状态：当前自动机位置、全局处理偏移、块边界残留数据、累计命中/输出上下文；这四类合起来才能保证跨块正确性
 - 每次追加输入都是“续跑”而不是“重跑”：先补齐上次残留，再继续状态转移，处理完后把最新状态写回会话，等待下个 chunk
 - 读取结果和重置状态分离：读结果只是看当前快照，不改变状态；只有显式清理才回到初始态，避免隐式副作用
 - 工程上的前提是匹配期状态要稳定：先确保自动机结构构建完成，再进入增量匹配，匹配过程中不并发修改词库
 - 本质上，这不是算法升级，而是把 AC 从“批处理函数”改造成“有状态执行器”
